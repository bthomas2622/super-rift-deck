## Project

Super Rift Deck — a static, client-side deckbuilder for the Riftbound TCG. Vanilla JS + Vite, deployed to GitHub Pages. No framework, no backend at runtime.

## Commands

```sh
npm install            # install deps (Vite is the only one)
npm run dev            # dev server at http://localhost:5173/super-rift-deck/
npm run build          # production build → dist/
npm run preview        # preview the built dist/
npm run fetch-cards    # repopulate public/data/ from the Riftcodex API (slow; hits network + downloads WebP images)
```

There is no test runner, linter, or formatter configured — do not invent commands for these.

## Vite layout quirks

`vite.config.js` sets `root: 'src'` and `publicDir: ../public`, with `base: '/super-rift-deck/'`. Consequences:

- `src/index.html` is the entry; asset paths in it are relative to `src/`.
- Anything under `public/` (notably `public/data/`) is served at the site root, e.g. `/super-rift-deck/data/cards.json`. The app fetches card JSON from this path — keep the base prefix in mind when adding fetch calls.
- `dist/` is written to the repo root, not inside `src/`.

## Architecture

Single-page app with no framework. `src/main.js` is the entry point and the single owner of mutable state:

- `allCards`, `sets`, `indexes` — loaded once from `public/data/*.json` (generated by `scripts/fetch-cards.mjs`).
- `filterState` — created by `createFilterState()` in `filters.js`.
- `deckState` — `{ legend, champion, mainDeck, runes, battlefields, sideboard, addToSideboard }` where the four collections are `Map<name, { card, count }>`.

Components in `src/components/` are plain modules that export `render*` functions. They receive the relevant slice of state plus callbacks from `main.js`; they do not own state themselves. To add a feature, add a component module, import it in `main.js`, and wire it up there.

Key component responsibilities:

- `filters.js` — filter state shape, filter UI, `applyFilters`, sort, and URL-param round-tripping (`filterStateToParams` / `filterStateFromParams`).
- `card-grid.js` — the browse grid.
- `deck-panel.js` — the deck sidebar list view.
- `deck-details.js` — analytics charts; also exports `computeRuneSplit` used elsewhere.
- `deck-validation.js` — enforces Riftbound construction rules (Domain Identity from Legend, Chosen Champion tag match, copy limits, Signature limits, Rune=12, Battlefields=3 unique).
- `deck-io.js` — deck import/export across multiple formats (Super Rift Deck, Riftbound.gg, PiltoverArchive, Rift Atlas, CardNexus). Sample files live in `exampleimportexport/deck/<format>/` and are the source of truth for format compatibility — consult them before changing parsers.
- `collection-io.js` — owned-card collection import/export (CSV formats: SRD, Riftbound.gg, PiltoverArchive, CardNexus). State shape `Map<shortId, { card, normal, foil }>`; persisted to `localStorage` under `riftbuilder_collection`. Samples in `exampleimportexport/collection/<format>/`.
- `deck-image.js` — PNG snapshot generation via canvas; depends on local WebP images under `public/data/images/`.
- `hand-simulator.js` — opening-hand draw + mulligan.

Deck state is persisted to `localStorage`; view switching (`card` / `deck` / `details` / `hand`) is handled in `main.js` via `setActiveView`.

## Card data pipeline

`scripts/fetch-cards.mjs` pulls from `https://api.riftcodex.com`, downloads card images as WebP at width 300 into `public/data/images/`, and writes `cards.json`, `sets.json`, `indexes.json`. It is run by the nightly `fetch-cards.yml` workflow and commits changes; running it locally is only needed when working on the fetch script itself or testing against fresh data.

## GitHub Actions

- `deploy.yml` — builds and deploys to GitHub Pages on push to `main`.
- `fetch-cards.yml` — nightly card-data refresh (cron + manual dispatch), commits diffs to `public/data/`.
